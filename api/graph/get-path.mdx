title: Get Path

## Endpoint

```
GET /graph/path
```

Finds a path between two songs in the listening graph using either shortest path (BFS) or strongest path (max-min Dijkstra) algorithms.

## Query Parameters

<ParamField query="user" type="string" required>
  The Last.fm username whose graph to query
</ParamField>

<ParamField query="from" type="string" required>
  The starting node - either a UUID or a SongKey (must contain `::`)
</ParamField>

<ParamField query="to" type="string" required>
  The destination node - either a UUID or a SongKey (must contain `::`)
</ParamField>

<ParamField query="algorithm" type="string" default="shortest">
  Path-finding algorithm to use:
  - `shortest`: Fewest hops (BFS)
  - `strongest`: Maximum minimum edge weight (bottleneck path)
</ParamField>

## Response

<ResponseField name="from" type="string">
  The SongKey of the starting node
</ResponseField>

<ResponseField name="to" type="string">
  The SongKey of the destination node
</ResponseField>

<ResponseField name="found" type="boolean">
  Whether a path was found between the two songs
</ResponseField>

<ResponseField name="algorithm" type="string">
  The algorithm used: `"shortest"` or `"strongest"`
</ResponseField>

<ResponseField name="path" type="array">
  Ordered list of nodes from start to end
  
  <Expandable title="PathStep properties">
    <ResponseField name="songKey" type="string">
      Canonical song identifier
    </ResponseField>
    
    <ResponseField name="name" type="string">
      Track name
    </ResponseField>
    
    <ResponseField name="artists" type="string[]">
      Artist names
    </ResponseField>
    
    <ResponseField name="edgeWeight" type="integer">
      Number of transitions from this node to the next (undefined for the last node)
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="hops" type="integer">
  Number of edges in the path (path.length - 1)
</ResponseField>

<ResponseField name="totalWeight" type="integer">
  Sum of all edge weights along the path
</ResponseField>

<ResponseField name="minEdgeWeight" type="integer">
  Minimum edge weight along the path (relevant for strongest path algorithm)
</ResponseField>

## Algorithm Details

### Shortest Path (BFS)

Finds the path with the **fewest hops** between two songs:
- Uses breadth-first search
- Ignores edge weights
- Optimal for finding "degrees of separation"
- Fast: O(V + E) complexity

**Use case**: "How many songs separate Radiohead and The Beatles in my listening history?"

### Strongest Path (Max-Min Dijkstra)

Finds the path that **maximizes the minimum edge weight** (the bottleneck/widest path):
- Modified Dijkstra's algorithm
- Finds the path where the weakest link is as strong as possible
- Optimal for finding "well-traveled" routes
- Slower: O(E log V) complexity

**Use case**: "What's the most frequently-traveled route from Song A to Song B?"

## Interpreting Results

### Edge Weights

Edge weights represent the number of times you listened to Song B immediately after Song A:
- High weights (>10): Very common transition
- Medium weights (3-10): Occasional transition
- Low weights (1-2): Rare transition

### Path Patterns

- **Direct path** (1 hop): Songs you frequently play back-to-back
- **Short path** (2-3 hops): Songs in the same listening context/mood
- **Long path** (4+ hops): Songs from different clusters/genres
- **No path**: Songs from disconnected components of your graph

## Use Cases

- **Playlist generation**: Create smooth transitions between songs
- **Music discovery**: Find unexpected connections between artists
- **Listening pattern analysis**: Understand how genres connect in your habits
- **Recommendation**: Suggest songs based on strongest paths from favorites

## Notes

<Note>
- Paths follow directed edges (respects listening order)
- Both UUIDs and SongKeys are accepted (SongKeys must contain `::`)
- If no path exists, the graph has disconnected components
- The strongest path algorithm prioritizes consistency over brevity
</Note>

<Warning>
For very large graphs (>10,000 nodes), path-finding may take several seconds. Consider implementing timeouts on the client side.
</Warning>
