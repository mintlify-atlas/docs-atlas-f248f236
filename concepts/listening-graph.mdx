## What is a Listening Graph?

A **listening graph** is a directed, weighted network where:

- **Nodes** represent individual songs
- **Edges** represent sequential transitions (song A played before song B)
- **Weights** indicate how often each transition occurred

<Info>
  Your listening history becomes a rich data structure revealing patterns, clusters, and connections that algorithmic recommendations miss.
</Info>

## Graph Structure

### Nodes (Songs)

Each song in the graph is identified by a **SongKey**:

```typescript
// From graph-server/src/graph/types.ts

/**
 * Canonical song identity: `lowercase(artist)::lowercase(track_name)`.
 * Used to match the same song across data sources.
 */
export type SongKey = `${string}::${string}`

export function toSongKey(artist: string, track: string): SongKey {
  return `${artist.toLowerCase().trim()}::${track.toLowerCase().trim()}`
}
```

This normalization allows the same song to be matched across:
- Last.fm scrobbles
- Spotify playlists
- Bandcamp imports
- YouTube streams

### Graph Node Structure

```typescript
export interface GraphNode {
  name: string
  artists: string[]
  albumName?: string
  lastfmUrl?: string
  mbid?: string  // MusicBrainz ID
  
  // Weighted edges
  next: Record<SongKey, number>      // Outgoing: song → count
  previous: Record<SongKey, number>  // Incoming: song → count
  
  totalPlays: number
  sources: ListeningSource[]  // ["lastfm", ...]
  sourcePlays?: Partial<Record<ListeningSource, number>>
  
  // Analysis results
  pageRank?: number
  clusterId?: number
  imageUrl?: string
  playDates: string[]  // ISO 8601 timestamps
  positions?: LayoutPositions  // Pre-computed x/y coordinates
}
```

### Edge Weights

Edges are weighted by frequency:

```typescript
// Example: Song A followed by Song B three times
nodeA.next["song-b::title"] = 3
nodeB.previous["song-a::title"] = 3
```

This creates a **bidirectional weighted graph** capturing:
- Forward listening patterns (what came next)
- Backward listening patterns (what led here)
- Strength of associations (how often)

## Graph Construction

The graph is built in phases:

<Steps>
  <Step title="Data Ingestion">
    Fetch raw data from all sources:
    
    ```bash
    # Fetch Last.fm scrobbles
    curl -X POST http://localhost:3001/pipeline/fetch/lastfm \
      -d '{"username":"your_username"}'
    ```
  </Step>
  
  <Step title="Normalization">
    Convert all tracks to canonical SongKeys:
    
    ```typescript
    const songKey = toSongKey("The Beatles", "Hey Jude")
    // Result: "the beatles::hey jude"
    ```
    
    This deduplicates the same song across sources.
  </Step>
  
  <Step title="Edge Creation">
    Process chronologically ordered scrobbles:
    
    ```typescript
    for (let i = 0; i < scrobbles.length - 1; i++) {
      const current = toSongKey(scrobbles[i].artist, scrobbles[i].track)
      const next = toSongKey(scrobbles[i + 1].artist, scrobbles[i + 1].track)
      
      // Create weighted edge
      graph.nodes[current].next[next] = 
        (graph.nodes[current].next[next] || 0) + 1
      graph.nodes[next].previous[current] = 
        (graph.nodes[next].previous[current] || 0) + 1
    }
    ```
  </Step>
  
  <Step title="Analysis">
    Compute PageRank, clusters, and statistics (see below)
  </Step>
  
  <Step title="Storage">
    Persist to PostgreSQL with per-user isolation:
    
    - `users` table
    - `nodes` table (UUID-keyed)
    - `edges` table (weighted)
    - `metadata` table
    - `pipeline_jobs` table
  </Step>
</Steps>

## PageRank Analysis

PageRank identifies the most "influential" songs in your listening graph:

```typescript
// From graph-server/src/analysis/pagerank.ts

export function computePageRank(
  graph: ListeningGraph,
  options: PageRankOptions = {}
): PageRankResult {
  const {
    dampingFactor: d = 0.85,
    convergenceThreshold = 0.0001,
    maxIterations = 100,
  } = options

  // Initialize all nodes with rank 1/N
  const keys = Object.keys(graph.nodes) as SongKey[]
  const n = keys.length
  const ranks = new Map<SongKey, number>()
  
  for (const key of keys) {
    ranks.set(key, 1 / n)
  }

  // Iterative algorithm
  while (iterations < maxIterations && !converged) {
    // Distribute rank along weighted edges
    for (const fromKey of keys) {
      const node = graph.nodes[fromKey]
      const outSum = Object.values(node.next).reduce((a, b) => a + b, 0)
      
      for (const [toKey, weight] of Object.entries(node.next)) {
        const contribution = (ranks.get(fromKey) * weight) / outSum
        newRanks.set(toKey, newRanks.get(toKey) + d * contribution)
      }
    }
    
    // Check convergence...
  }
  
  // Write ranks back to graph
  for (const key of keys) {
    graph.nodes[key].pageRank = ranks.get(key)
  }
}
```

**What PageRank reveals:**
- Songs you frequently transition TO from many different songs
- "Hub" songs that bridge different parts of your library
- Core songs in your rotation vs. peripheral ones

<Tip>
  High PageRank songs aren't necessarily your most-played - they're your most *connected* songs
</Tip>

## Cluster Detection

Clusters reveal natural groupings in your listening:

- **Genre clusters** - Similar music styles
- **Mood clusters** - Songs listened to in similar contexts
- **Artist clusters** - Catalog exploration patterns
- **Temporal clusters** - Songs from the same listening period


## Visualization

The graph frontend renders your listening graph using Sigma.js:

```typescript
// From site/src/components/GraphView.tsx
import { SigmaContainer, useLoadGraph } from "@react-sigma/core"

function GraphInner() {
  const { graph, state, error } = useGraphData()
  const loadGraph = useLoadGraph()
  
  useEffect(() => {
    if (!graph) return
    loadGraph(graph)  // Load into Sigma
    
    // Auto-focus a random node on startup
    const nodes = graph.nodes()
    const randomKey = nodes[Math.floor(Math.random() * nodes.length)]
    onSelectNode({ key: randomKey, attrs: graph.getNodeAttributes(randomKey) })
  }, [graph])
  
  return <GraphEvents onSelectNode={onSelectNode} />
}
```

## API Access

All graph endpoints require `?user=<username>` parameter:

| Endpoint | Description |
|----------|-------------|
| `GET /graph` | Full graph (paginated with `?limit=&offset=`) |
| `GET /graph/node/:id` | Single node by UUID |
| `GET /graph/neighbors/:id` | Node plus immediate neighbors |
| `GET /graph/stats` | Summary counts + metadata |
| `GET /graph/analysis` | PageRank top N (optional `?topN=`) |
| `GET /graph/path` | Path query (`?from=&to=&algorithm=`) |

<Warning>
  Graph endpoints return `400` if `?user=` is missing and `404` if the user is unknown
</Warning>
